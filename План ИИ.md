# План ИИ Кима: Пошаговый Туториал Для Новичка

Версия: 2026-04-15  
Целевая платформа: Windows + Godot 4.6.2



## Что получится в конце

После прохождения этого туториала у тебя будет ИИ Ким, который:

1. Ведет диалог через локальную LLM (NobodyWho + GGUF).
2. Помнит предпочтения игрока и подстраивает стиль общения.
3. Генерирует идеи для создания объектов.
4. Создает объекты по структурированным командам.
5. Учится на лайках/дизлайках.
6. Сохраняет состояние между перезапусками.

## Требования

### ПО

1. Godot Engine 4.6.2 (или 4.5+).
2. Git (желательно, для контроля изменений).
3. Доступ в интернет для скачивания плагинов и модели.

### Железо (практические пороги)

1. Для 1B моделей: CPU 4+ ядер, RAM 8 GB, желательно GPU 4 GB VRAM.
2. Для 4B моделей (рекомендуется): CPU 6+ ядер, RAM 16 GB, желательно GPU 8 GB VRAM.
3. Для 14B моделей: CPU 8+ ядер, RAM 32 GB+, желательно GPU 12-16 GB VRAM.

Рекомендованный старт: `Qwen3 4B GGUF`.

## Запуск MVP: 12 шагов

### Шаг 1. Проверка проекта и Autoload

`Где делать:` Godot Editor + `project.godot`  
`Что делать:` Убедиться, что проект открывается, а `KimCore` подключен как Autoload.  
`Клики/команды:`

```text

Проверь наличие: Name=KimCore, Path=res://scripts/ai_modules/kim_core.gd
```

`Ожидаемый результат:` Проект запускается без parse errors, `KimCore` есть в списке Autoload.  
`Если не работает:` Исправь синтаксис скриптов по ошибке в Output, затем перезапусти проект.

`Клики/команды:`

AssetLib -> Search: Godot XR Tools -> Install (optional)

`Ожидаемый результат:` В Plugins нужные плагины имеют статус Enabled.  
`Если не работает:` Проверь версию Godot и совместимость плагина, при необходимости поставь версию из архива Godot.

## План тестирования

1. Чистая машина: новичок проходит шаги и получает первый ответ Кима.
2. Без модели: fallback-режим работает и дает понятную диагностику.
3. Два клиента: маршрут client -> server -> Kim -> RPC response валиден.
4. Перезапуск: профиль и веса восстанавливаются из `user://`.
5. FAQ-проверка: типовые ошибки решаются только по этому документу.

## FAQ: типовые ошибки и фиксы

### 1) `Class "KimCore" hides an autoload singleton`

Причина: в autoload и `class_name` используется одно имя.  
Фикс: убрать `class_name KimCore` из `kim_core.gd` или переименовать класс.

### 2) `Warning treated as error (Variant inference)`

Причина: включен режим строгой компиляции, а переменная без явного типа.  
Фикс: проставить явные типы (`Object`, `Script`, `Variant`, `Dictionary`, `Array`) в проблемных строках.

### 3) `unsupported feature: forward_plus`

Причина: железо/драйвер не поддерживает требуемый метод рендера.  
Фикс: убрать `config/features=PackedStringArray("forward_plus")` и переключить рендерер на совместимый.

### 4) `backend missing` в `personality_module`

Причина: не найден `NobodyWhoChat`/`NobodyWhoModel` или плагин не активен.  
Фикс: проверить `Project Settings -> Plugins` и узлы/поля в `personality_module.tscn`.

### 5) GGUF не загружается

Причина: неверный формат/путь или слишком тяжелая модель для памяти.  
Фикс: проверить расширение `.gguf`, путь `res://models/...`, начать с квантизованной 4B модели.

### FAQ: мультимодальный трек

### Почему нужен Godot .NET

Прямой C# gRPC-клиент стабилен для real-time интеграции и проще в сопровождении, чем обходные решения.

### Как совместить выбор Q8_K_XL с основным HF/vLLM путем

HF FP/BF16 остается основным путем для дообучения и мультимодального качества. `Q8_K_XL` использовать как optional runtime-профиль для экономии памяти.

### Когда брать YAMNet/AST вместо BEATs+CLAP

`YAMNet` — быстрый старт и простая интеграция. `AST` — когда важнее качество классификации и есть ресурсы. `BEATs+CLAP` — базовый рекомендуемый трек в этом документе.

### Как безопасно откатываться на MVP-текстовый контур

Держи feature-flag мультимодальности, health-check, таймауты и rollback-путь. При деградации отключай perception и оставляй рабочими `GameServer.send_text_to_kim(...)` и `KimCore.activate_module("personality").send_message(...)`.

### Test Plan (для мультимодального раздела)

1. Проверить, что новый раздел стоит сразу после `Чеклист готовности`, а MVP-часть не изменена.
2. Проверить, что все 12 шагов содержат формат из 5 полей (`Где делать -> Что делать -> Клики/команды -> Ожидаемый результат -> Если не работает`).
3. Проверить, что старые контракты сохранены, а мультимодальные контракты добавлены.
4. Проверить, что документ явно описывает основной мультимодальный режим и fallback на MVP-текстовый контур.
5. Проверить, что путь `Qwen3-VL + BEATs/CLAP + gRPC` можно пройти без внешнего поиска и получить ожидаемые результаты.

### Assumptions (мультимодальный раздел)

1. Целевая среда: Windows, Godot 4.6.2 `.NET`, `.NET SDK 8+`.
2. Основной vision runtime: `Transformers + vLLM`; основной checkpoint: HF FP/BF16.
3. `Q8_K_XL` фиксируется как optional runtime-профиль, не как основной путь дообучения.
4. Базовый sound-event модуль: `BEATs+CLAP`; `YAMNet/AST` — альтернативы.
5. Глубина: полный путь M0-M2 для интеграции и дообучения.

### Гейт перед R&D

В этот раздел переходить только если:

1. Закрыт весь MVP-чеклист выше.
2. Есть воспроизводимые тесты из раздела `План тестирования`.

### Единый микрошаблон внедрения (для каждой технологии)

1. `Что это`: короткое определение.
2. `Зачем Киму`: какой прирост для проекта.
3. `Где в стеке`: в каком контуре внедряется (train/infer/serving).
4. `Как сделать`:
   - `Предусловия`
   - `M0` (минимальный запуск)
   - `M1` (интеграция в Кима)
   - `M2` (метрика успеха)
   - `Stop-критерий и fallback`
5. `Критерий готовности`: что считаем "сделано".
6. `Риски/ограничения`: когда не внедрять.

### Трек A. Энергоэффективный интеллект

#### 1) Neuromorphic computing

`Что это:` вычисления на архитектурах, вдохновленных биологическим мозгом (событийные/импульсные схемы).  
`Зачем Киму:` потенциально снизить энергопотребление и latency для edge-сценариев.  
`Где в стеке:` отдельная train/infer лаборатория вне Godot, затем перенос результата через distilled/adapter модель.  
`Как сделать:`  
`Предусловия:` отдельное Python-окружение, датасет диалогов Кима, baseline-метрики текущей LLM.  
`M0:` поднять эксперимент на CPU/GPU-эмуляции нейроморфного pipeline (без интеграции в Godot).  
`M1:` обучить/дистиллировать компактную модель, экспортировать в обычный LLM-совместимый формат для inference.  
`M2:` сравнить против baseline: задержка, энергозатраты, качество ответов (минимум не хуже baseline-1sigma).  
`Stop-критерий и fallback:` если качество падает >10% по ключевым тестам, откат на классический pipeline + оставить R&D-ветку отдельно.  
`Критерий готовности:` стабильный прирост энергоэффективности при приемлемом качестве в офлайн-тестах.  
`Риски/ограничения:` высокая исследовательская сложность, редкий tooling для прямой интеграции в Godot.

#### 2) Спайковые нейронные сети (SNN)

`Что это:` сети, работающие импульсами (spikes), часто применяются в нейроморфных системах.  
`Зачем Киму:` уменьшить compute на простых/частых задачах и исследовать event-driven обработку.  
`Где в стеке:` предмодуль обработки сигналов/признаков перед LLM или в отдельном student-модуле.  
`Как сделать:`  
`Предусловия:` готовый диалоговый корпус Кима, baseline latency и accuracy.  
`M0:` собрать SNN-прототип для одной задачи (например, topic/tone классификация).  
`M1:` подключить выход SNN как feature input в `analyst_module` и профиль игрока.  
`M2:` проверить, что классификация не хуже baseline, а latency меньше.  
`Stop-критерий и fallback:` если нет выигрыша по latency/энергии, оставить классический классификатор.  
`Критерий готовности:` SNN-подмодуль стабильно улучшает один практический контур (аналитика/триггеры).  
`Риски/ограничения:` трудно масштабировать до полного генератора текста.

#### 3) Биполярные морфологические нейроны

`Что это:` специализированные нейронные элементы для морфологических операций (max/min-подобные преобразования).  
`Зачем Киму:` устойчивые и интерпретируемые признаки для поведения/аналитики.  
`Где в стеке:` feature engineering в `analyst_module` и scoring в active learning.  
`Как сделать:`  
`Предусловия:` четко определенные признаки (темы, тон, интенсивность взаимодействия).  
`M0:` внедрить как отдельный экспериментальный классификатор признаков.  
`M1:` смешать с текущей аналитикой через weighted ensemble.  
`M2:` измерить прирост F1/MAE по профилированию игрока.  
`Stop-критерий и fallback:` если интерпретируемость выше, но метрики хуже, использовать только для explainability-режима.  
`Критерий готовности:` полезен как дополнительный сигнал, не ломает основной pipeline.  
`Риски/ограничения:` нишевая тема, ограниченный production-tooling.

### Трек B. Эффективное обучение данных

#### 4) Active Learning

`Что это:` стратегия отбора самых "полезных" примеров для разметки и дообучения.  
`Зачем Киму:` быстрее улучшать модель на реальных диалогах при меньшей стоимости разметки.  
`Где в стеке:` контур `сбор логов -> отбор -> доразметка -> дообучение`.  
`Как сделать:`  
`Предусловия:` логирование диалогов и меток качества ответа.  
`M0:` запустить random baseline отбор + ручная разметка пакета.  
`M1:` заменить random на uncertainty-based sampling и сравнить.  
`M2:` добиться лучшего качества при том же бюджете разметки.  
`Stop-критерий и fallback:` если uncertainty не дает прироста, временно вернуться на stratified random.  
`Критерий готовности:` на одном и том же бюджете разметки active learning лучше baseline.  
`Риски/ограничения:` требует дисциплины в логировании и разметке.

#### 5) ATGen (фреймворк активного обучения)

`Что это:` toolkit для active learning в генеративных задачах.  
`Зачем Киму:` ускорить эксперименты по стратегиям отбора данных.  
`Где в стеке:` отдельный training pipeline вне Godot (Python), результаты импортируются в модель Кима.  
`Как сделать:`  
`Предусловия:` Python 3.10+, GPU для обучения, подготовленный датасет диалогов Кима.  
`M0:` поднять ATGen на небольшом subset и прогнать random/huds/hadas стратегии.  
`M1:` подключить экспорт лучших примеров обратно в цикл дообучения student-модели Кима.  
`M2:` фиксировать выигрыш в метриках при том же объеме разметки.  
`Stop-критерий и fallback:` если фреймворк нестабилен, оставить саму методологию AL и реализовать простой custom pipeline.  
`Критерий готовности:` ATGen дает воспроизводимый и автоматизированный AL-цикл.  
`Риски/ограничения:` проект может быть alpha-уровня, нужна проверка совместимости.

#### 6) Вероятностный метод обучения (Bayesian uncertainty sampling)

`Что это:` вероятностная оценка уверенности модели для отбора данных в active learning.  
`Зачем Киму:` фокус на "сомнительных" примерах, которые быстрее улучшают поведение.  
`Где в стеке:` scoring-слой в контуре AL + calibration.  
`Как сделать:`  
`Предусловия:` сбор logits/score confidence, валидационный набор.  
`M0:` ввести uncertainty score (entropy, margin, variance) для каждого примера.  
`M1:` добавить калибровку вероятностей (temperature scaling/isotonic) перед отбором.  
`M2:` показать, что uncertainty sampling > random sampling по приросту качества за N итераций.  
`Stop-критерий и fallback:` если uncertainty шумный, смешивать с diversity sampling в гибридной схеме.  
`Критерий готовности:` устойчивый прирост качества на фиксированном annotation budget.  
`Риски/ограничения:` плохая калибровка может портить отбор.

### Трек C. Сжатие и адаптация модели

#### 7) Knowledge Distillation

`Что это:` перенос поведения из teacher-модели в более легкую student-модель.  
`Зачем Киму:` снизить нагрузку без сильной потери качества.  
`Где в стеке:` train pipeline; inference остается в текущем Godot/NobodyWho контуре.  
`Как сделать:`  
`Предусловия:` teacher=`Qwen3-4B`, student=`Qwen3-1.7B`, набор промптов Кима.  
`M0:` собрать пары `prompt -> teacher_output` и обучить student на imitation loss.  
`M1:` подключить student как основную модель в локальный inference и прогнать regression suite.  
`M2:` зафиксировать снижение latency/RAM при acceptable quality drop (например <=5%).  
`Stop-критерий и fallback:` если деградация качества выше порога, оставить 4B для сложных запросов + 1.7B для быстрых.  
`Критерий готовности:` student способен заменить teacher в большинстве типовых диалогов.  
`Риски/ограничения:` риск копирования ошибок teacher и потери long-tail знаний.

#### 8) Adaptive fine-tuning / BiDoRA

`Что это:` параметр-эффективная донастройка (цель: BiDoRA, fallback: DoRA/LoRA).  
`Зачем Киму:` быстро адаптировать стиль/поведение без полного переобучения модели.  
`Где в стеке:` PEFT-обучение вне Godot, затем подключение адаптера к inference pipeline.  
`Как сделать:`  
`Предусловия:` подготовленный instruction dataset по персонажу Ким, контрольный бенч ответов.  
`M0:` старт с DoRA/LoRA baseline для воспроизводимости.  
`M1:` включить BiDoRA (если tooling доступен) и сравнить с baseline при одинаковом бюджете.  
`M2:` выбрать метод по Pareto: качество/скорость/память.  
`Stop-критерий и fallback:` если BiDoRA нестабилен или несовместим, зафиксировать DoRA/LoRA как production baseline.  
`Критерий готовности:` адаптер улучшает стиль Кима и не ухудшает базовую безопасность.  
`Риски/ограничения:` совместимость библиотек и формат адаптеров.

### Трек D. Инфраструктура и производительность

#### 9) AutoML / HPO

`Что это:` автоматический подбор гиперпараметров обучения/инференса.  
`Зачем Киму:` быстрее находить рабочие конфиги без ручного перебора.  
`Где в стеке:` training scripts + evaluation harness.  
`Как сделать:`  
`Предусловия:` формализованная целевая функция (качество, latency, RAM, cost).  
`M0:` запустить маленький search-space (lr, rank, batch, epochs).  
`M1:` расширить search и внедрить early stopping/pruning.  
`M2:` зафиксировать выигрыш по целевой функции относительно ручного baseline.  
`Stop-критерий и fallback:` если поиск нестабилен/дорог, оставить curated presets для 1.7B/4B.  
`Критерий готовности:` воспроизводимый “лучший” конфиг с логами и seed.  
`Риски/ограничения:` cost explosion при широком search-space.

#### 10) FP8 training

`Что это:` обучение/дообучение в FP8 для ускорения и снижения потребления памяти на поддерживаемом железе.  
`Зачем Киму:` ускорить этапы донастройки крупных конфигов.  
`Где в стеке:` training-only контур, не обязательный для runtime в Godot.  
`Как сделать:`  
`Предусловия:` GPU/стек с поддержкой FP8 (например, соответствующий backend и framework).  
`M0:` пилот на коротком прогоне и проверка численной стабильности.  
`M1:` включить mixed precision pipeline + мониторинг loss-scale/NaN.  
`M2:` сравнить wall-clock и качество относительно BF16/FP16 baseline.  
`Stop-критерий и fallback:` при нестабильной сходимости откат на BF16.  
`Критерий готовности:` ощутимый выигрыш скорости без деградации ключевых метрик.  
`Риски/ограничения:` аппаратная/библиотечная зависимость, сложная диагностика нестабильностей.

#### 11) Aegaeon (Alibaba Cloud)

`Что это:` multi-model serving подход с токен-уровневым scheduling для повышения утилизации GPU в облаке.  
`Зачем Киму:` дешевле и эффективнее обслуживать несколько моделей/вариантов Кима в cloud-режиме.  
`Где в стеке:` cloud-only serving слой (опционально, не для локального MVP).  
`Как сделать:`  
`Предусловия:` cloud-окружение Alibaba, набор моделей и SLA по задержке/стоимости.  
`M0:` пилотный стенд с 2-3 моделями и метриками utilization/latency/cost.  
`M1:` внедрить routing policy между локальным runtime и cloud inference профилями.  
`M2:` подтвердить снижение стоимости и рост throughput на целевой нагрузке.  
`Stop-критерий и fallback:` если интеграция слишком дорогая, использовать обычный single-model serving или vLLM/TGI-профиль.  
`Критерий готовности:` cloud-профиль дает экономический выигрыш без деградации SLA.  
`Риски/ограничения:` vendor lock-in, сеть, требования к эксплуатации.

### FAQ-дополнения по R&D

#### Почему SNN не в MVP

Потому что MVP ориентирован на быстрый рабочий диалоговый контур в Godot, а SNN требует отдельного исследовательского пайплайна и не дает гарантированного выигрыша в первом релизе.

#### Когда FP8 дает выигрыш, а когда нет

Дает выигрыш при совместимом железе и стабильном backend. Не дает, если обучение становится численно нестабильным и приходится откатываться на BF16/FP16.

#### Что делать, если BiDoRA tooling недоступен

Использовать DoRA/LoRA как production baseline, а BiDoRA оставить как экспериментальную ветку без блокировки релиза.

#### Как использовать Aegaeon-концепцию без полного перехода в Alibaba Cloud

Взять идею multi-model pooling/routing: локально держать 1.7B как fast-path и облачную 4B+ как quality-path через policy-роутинг, без обязательного полного переноса в Alibaba Cloud.

## Optional: VR-ветка после MVP

1. Установить `Godot XR Tools` и `OpenXR Vendors Plugin v5`.
2. Включить OpenXR в `Project Settings -> XR`.
3. Создать VR-сцену с `XROrigin3D`, `XRCamera3D`, контроллерами.
4. Подключить `FunctionTeleport` и `FunctionPickup`.
5. Синхронизировать VR-аватар через `MultiplayerSynchronizer`.

Правило: сначала закрыть MVP-чеклист, и только потом переходить в VR.
